# QuickRec v1.4.x 诊断日志 / 错误导出能力 PRD

> 版本：v1.4.x 后续版本  
> 类型：正式推进型 PRD  
> 状态：已验收通过 / 可发布  
> 来源：`doc/mypm-idea-pool-2026-07-08.md` IDEA-003  
> 方案：标准导出方案（复制诊断信息 + 打开日志目录 + 导出诊断文件）

## 1. 需求状态

### 1.1 PRD 类型

本需求为 **正式推进型 PRD**。

判断依据：

- v1.4 已完成稳定性与工程化收口，后续需要提高错误定位效率。
- 历史测试与发布验证中出现过 FFmpeg 启动失败、cv2 缺失导致 0 帧、音频混合失败、窗口捕获异常、保存失败等问题。
- 当前项目已有分散日志，但没有统一的复制 / 导出诊断信息入口。
- 该能力直接服务 Full v1.4.x 的稳定性、可维护性和用户信任。

### 1.2 当前确认

- 本次采用方案 2：标准导出方案。
- 版本范围：QuickRec Full v1.4.x 后续版本。
- 诊断文件默认存放在视频保存路径下的新建诊断文件夹中。
- 诊断目录允许用户自定义。
- 当前诊断只保存在本地。
- 当前版本无需脱敏。

### 1.3 当前假设

- Lite 分支暂不作为本 PRD 的主要开发目标；后续可复用能力，但不要求 Lite v0.1 必须进入。
- 本功能不改变录制核心链路，只补充日志持久化、诊断信息生成和用户导出入口。
- 诊断信息主要服务用户向开发者反馈问题，不做自动修复。

## 2. 方案收敛

### 2.1 候选方案回顾

| 方案 | 内容 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- | --- |
| 方案 1：最小复制方案 | 只提供复制诊断信息、打开日志目录 | 成本最低 | 不便于携带完整日志 | 不采用 |
| 方案 2：标准导出方案 | 复制诊断信息、打开日志目录、导出诊断文件 | 排查价值高，仍保持轻量 | 需要定义日志目录、文件格式、失败反馈 | 采用 |
| 方案 3：诊断面板方案 | 独立诊断窗口展示状态和错误 | 信息完整 | 接近复杂诊断中心，成本偏高 | 延后 |

### 2.2 当前版本目标

让用户在录制失败、音频异常、FFmpeg 异常、保存失败、窗口捕获异常时，可以通过轻量入口快速复制或导出本地诊断信息，降低排查成本。

### 2.3 优先级策略

本版本按以下原则排序：

```text
优先级 = 错误定位价值 + 实现成本可控 + 对现有录制链路低侵入
       - UI 复杂度 - 隐私/权限风险 - 打包风险
```

| 需求 | 优先级 | 进入版本 | 理由 |
| --- | --- | --- | --- |
| 日志文件持久化 | P0 | v1.4.x | 没有日志文件，导出能力无基础 |
| 复制诊断信息 | P0 | v1.4.x | 最小可用反馈路径 |
| 打开日志目录 | P0 | v1.4.x | 用户可直接找到本地日志 |
| 导出诊断文件 | P1 | v1.4.x | 便于一次性反馈排查信息 |
| 诊断目录配置 | P1 | v1.4.x | 用户已确认允许自定义 |
| 独立诊断状态面板 | P3 | 延后 | 超出最小版本 |
| 云上传 / 自动修复 | 不做 | 不进入 | 明确非目标 |

### 2.4 本次范围

- 新增本地日志文件持久化。
- 新增诊断信息生成能力。
- 新增复制诊断信息。
- 新增打开日志目录。
- 新增导出诊断文件。
- 新增诊断目录配置项。
- 覆盖 FFmpeg、音频、录制、保存、窗口捕获等关键错误信息。

### 2.5 延后项

- 独立诊断中心或诊断面板。
- 自动检测环境并给出修复建议。
- 诊断信息脱敏规则。
- 云上传、Issue 自动创建、远程反馈。
- 按问题类型生成结构化修复指南。

### 2.6 不做项

- 不做云上传。
- 不做复杂诊断中心。
- 不做自动修复。
- 不收集隐私信息。
- 不自动发送任何本地文件。
- 不改变当前录制、编码、音频混流主链路。

## 3. 背景与问题

QuickRec 是本地桌面录屏工具，失败场景高度依赖用户机器环境。典型问题包括：

- FFmpeg 不存在、启动失败、编码失败。
- dxcam 捕获失败、窗口捕获失败、窗口关闭或最小化。
- 音频设备不可用、系统声或麦克风初始化失败、音频混合失败。
- 保存路径不存在、磁盘空间不足、最终文件移动失败。
- 打包环境中依赖缺失或路径不正确。

当前项目已有大量 `logger.info / warning / error`，但主要输出到运行时日志或控制台，用户遇到问题时无法稳定地找到、复制和导出排查信息。

## 4. 目标

### 4.1 用户目标

用户在录制异常后，可以不用理解技术细节，也能快速把本地排查信息交给开发者。

### 4.2 产品目标

提升 QuickRec Full v1.4.x 的可诊断性和用户信任，减少“无法复现 / 信息不足”的沟通成本。

### 4.3 工程目标

建立轻量诊断基础设施，为后续 Full 的质量诊断、捕获后端研究和特殊窗口兼容性问题排查提供统一材料。

## 5. 开发前验收与指标

### 5.1 验收目标

开发完成后，用户应能在本地完成三类操作：

1. 复制诊断信息到剪贴板。
2. 打开日志目录。
3. 导出诊断文件到诊断目录。

并且在典型失败场景中，诊断信息能包含足够排查问题的关键上下文。

### 5.2 验收标准

| 编号 | 场景 | 前置条件 | 操作 | 预期结果 |
| --- | --- | --- | --- | --- |
| D1 | 复制诊断信息 | 应用已启动 | 点击“复制诊断信息” | 剪贴板写入纯文本诊断摘要，并显示成功反馈 |
| D2 | 打开日志目录 | 应用已启动 | 点击“打开日志目录” | 系统资源管理器打开诊断目录 |
| D3 | 导出诊断文件 | 应用已启动 | 点击“导出诊断文件” | 在诊断目录生成 `.txt` 诊断文件，并显示文件路径 |
| D4 | FFmpeg 异常 | 模拟 FFmpeg 路径缺失或启动失败 | 启动录制后导出诊断 | 诊断信息包含 FFmpeg 路径、错误阶段、错误原因 |
| D5 | 音频异常 | 模拟系统声或麦克风不可用 | 录制后导出诊断 | 诊断信息包含请求音频源、最终音频源、降级原因 |
| D6 | 保存失败 | 模拟输出目录不可写或最终化失败 | 停止录制后导出诊断 | 诊断信息包含保存路径、session 目录、最终化错误 |
| D7 | 窗口捕获异常 | Full 窗口录制启动失败 | 导出诊断 | 诊断信息包含 hwnd、title、mode、stage、reason |
| D8 | 诊断目录自定义 | 修改诊断目录并保存 | 导出诊断文件 | 文件保存到自定义目录 |
| D9 | 诊断目录不可写 | 将诊断目录设为不可写路径 | 导出诊断文件 | 给出失败反馈，不崩溃，不影响录制 |

### 5.3 日志 / 人工检查

需要人工检查：

- 诊断文件可打开、编码为 UTF-8。
- 剪贴板文本可粘贴到文本编辑器。
- 资源管理器能打开诊断目录。
- 录制失败后用户仍能操作诊断入口。
- 打包产物中诊断功能可用。

需要日志检查：

- `diagnostic copied`
- `diagnostic directory opened`
- `diagnostic exported`
- `diagnostic export failed`
- 录制异常相关日志仍保留原始错误阶段。

### 5.4 通过标准

- 自动化测试覆盖诊断信息生成、目录配置、导出失败、日志文件路径。
- 手动验证至少覆盖复制、打开目录、导出文件三个主流程。
- 至少构造一个 FFmpeg 异常和一个音频异常，诊断信息中可定位原因。
- 打包产物中功能可用。
- 原有全屏 / 区域 / 窗口录制入口不因诊断能力回归失败。

### 5.5 不通过处理

- 如果诊断导出失败但影响录制：阻断发布，必须修复。
- 如果诊断文件缺少关键错误信息：补日志字段后重新验收。
- 如果仅自定义目录失败：可降级回默认诊断目录，但必须提示用户。
- 如果打包产物不可用：回到 PyInstaller spec 检查文件路径和权限。

### 6.5 配置

新增配置项：

| 配置项 | 类型 | 默认值 | 说明 |
| --- | --- | --- | --- |

## 8. 用户场景

### 8.1 录制启动失败

用户点击录制后提示“录制启动失败”。用户右键托盘，点击“导出诊断文件”，将生成的诊断文件发给开发者。

### 8.2 音频异常

用户选择“两者都有”，但最终视频没有麦克风声音。用户点击“复制诊断信息”，复制内容中包含请求音频源、最终音频源和降级原因。

### 8.3 保存失败

用户停止录制后提示保存失败。用户打开日志目录，查看最近日志和诊断文件。

### 8.4 窗口捕获异常

用户使用窗口录制时目标窗口无法录制。诊断文件中包含窗口句柄、标题、捕获阶段和失败原因。

## 9. 功能需求

### 9.1 日志文件持久化

- 需求类型：后端 / 日志系统。
- 触发条件：应用启动。
- 入口：`main.py` 初始化日志。
- 主流程：
  1. 读取配置中的诊断目录。
  2. 如果目录不存在，尝试创建。
  3. 将 QuickRec 日志写入 `quickrec.log`。
  4. 控制台日志保留现有行为。
- 异常流程：
  - 诊断目录创建失败时，降级到临时目录或 AppData 下默认目录，并记录 warning。
- 状态变化：
  - 新增日志文件。
  - 最近错误摘要可被诊断服务读取。
- 数据来源：
  - `ConfigManager`
  - Python logging

### 9.2 诊断信息生成

- 需求类型：后端 / 数据。
- 触发条件：用户点击复制或导出。
- 入口：托盘菜单或设置页按钮。
- 生成内容：
  - QuickRec 版本 / 分支口径。
  - Python 版本。
  - Windows 版本。
  - 是否打包运行。
  - 保存路径。
  - 诊断目录。
  - 当前录制状态。
  - 最近录制模式。
  - 最近输出路径。
  - FFmpeg 路径是否存在。
  - 音频源配置、音频预检结果、降级原因。
  - 最近窗口录制诊断信息。
  - 最近错误日志摘要。
  - 最近 100 行日志。
- 异常流程：
  - 某字段不可用时显示 `unknown`，不得抛出导致 UI 崩溃。

### 9.3 复制诊断信息

- 需求类型：前端 / UI / 日志系统。
- 入口：托盘菜单、设置页按钮。
- 用户操作：点击“复制诊断信息”。
- 主流程：
  1. 生成诊断摘要文本。
  2. 写入系统剪贴板。
  3. 显示“诊断信息已复制”。
- 失败反馈：
  - 剪贴板写入失败时提示“复制失败，请导出诊断文件”。
- 取消 / 关闭：
  - 设置页中点击取消不影响已经复制的诊断信息。

### 9.4 打开日志目录

- 需求类型：前端 / 文件系统。
- 入口：托盘菜单、设置页按钮。
- 用户操作：点击“打开日志目录”。
- 主流程：
  1. 确保诊断目录存在。
  2. 调用资源管理器打开目录。
- 失败反馈：
  - 目录创建失败或打开失败时提示“无法打开日志目录”。
- 取消 / 关闭：
  - 打开资源管理器后，QuickRec 设置页状态不变。

### 9.5 导出诊断文件

- 需求类型：前端 / 后端 / 文件存储。
- 入口：托盘菜单、设置页按钮。
- 用户操作：点击“导出诊断文件”。
- 主流程：
  1. 生成诊断文本。
  2. 保存为 `diagnostic_YYYYMMDD_HHMMSS.txt`。
  3. 写入诊断目录。
  4. 显示导出成功与文件路径。
- 失败反馈：
  - 写入失败时提示“导出失败，请检查诊断目录权限”。
- 取消 / 关闭：
  - 导出操作不弹出必须确认的 modal；设置页关闭不影响已导出的文件。

### 9.6 诊断目录配置

- 需求类型：配置 / UI。
- 入口：设置页“诊断”分组。
- 主流程：
  1. 默认显示 `<save_path>/QuickRecDiagnostics`。
  2. 点击“浏览...”选择自定义目录。
  3. 点击“保存”后写入配置。
- 分支流程：
  - 如果用户未自定义诊断目录，保存路径变化后诊断目录跟随变化。
  - 如果用户自定义诊断目录，保存路径变化不覆盖诊断目录。
- 失败反馈：
  - 选择目录为空时保持原值。
  - 保存时目录不可写，提示但不阻止其他设置保存；诊断导出时再降级或失败提示。

## 10. 交互与文案要求

### 10.1 文案

| 场景 | 文案 |
| --- | --- |
| 复制成功 | 诊断信息已复制 |
| 复制失败 | 复制失败，请导出诊断文件 |
| 打开目录失败 | 无法打开日志目录 |

### 10.2 UI 要求

- 诊断入口应保持轻量，不做大面积说明文字。
- 设置页中“诊断”分组放在快捷键之后。
- 按钮横向排列，避免设置窗口高度明显膨胀。
- 成功 / 失败反馈优先使用设置页状态提示或 Toast，不强制弹阻塞弹窗。

## 13. 风险与开放问题

### 13.1 风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 诊断目录不可写 | 导出失败 | 失败提示，允许用户自定义目录 |
| 日志文件过大 | 占用磁盘 | 首版可限制导出最近 100 行，后续再做滚动日志 |
| 诊断入口让 UI 变重 | 影响轻量感 | 放在设置页分组和托盘菜单，不做独立面板 |
| 日志字段不足 | 排查仍困难 | 验收时构造典型异常检查字段 |
| 打包路径差异 | 打包后导出失败 | 加打包产物人工验收 |

### 13.2 开放问题

- 是否需要在 Full 和 Lite 之间复用同一套诊断模块，还是 Full 先实现后再移植 Lite。
- 是否需要日志滚动策略，例如最多保留 7 天或 5MB。
- 是否在录制失败 Toast 中追加“复制诊断信息”动作。该能力依赖通知动作支持，首版可延后。
