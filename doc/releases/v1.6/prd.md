# QuickRec Full v1.6 轻量素材库基础版 PRD

> 版本：v1.6
> 类型：正式推进型 PRD
> 状态：需求已确认，功能实现与自动化验证完成，待 GUI 手动验收
> 产品线：QuickRec Full
> 原型：`doc/prototypes/product-prototype/full.html`（已同步 v1.6 并完成自动验证）
> 来源：`doc/archive/ideas/mypm-idea-pool-v1.6-2026-07-10.md`、v1.5 PRD、v1.5 验收与发布资料、当前代码结构

## 1. 需求状态

### 1.1 PRD 类型

本需求为 **正式推进型 PRD**。

判断依据：

- QuickRec Full v1.5 已发布并建立本地录制历史、最近录制窗口和统一入口。
- 当前历史索引随视频保存路径分散存放，切换路径会切换历史视图，无法形成稳定素材入口。
- v1.5 已存在稳定素材 ID、原子写入和缺失状态等基础能力，可增量演进，无需推倒重来。
- 用户已确认中央索引、迁移、恢复、重建、重新定位、容量、删除、原型和质量门禁等关键产品决策。
- 本版边界已收敛为一个产品主线和两个支撑模块，不包含完整工作台、AI 或捕获后端替换。

### 1.2 版本定位

v1.6 是 QuickRec Full 从“最近录制列表”进入“跨路径轻量素材库”的基础版本。

本版解决的是素材身份、集中索引、数据恢复和跨保存路径管理，不一次性交付完整创作者工作台。项目管理、内嵌预览、基础编辑和导出队列属于后续版本；AI 只保留兼容数据边界，不进入本版实现。

### 1.3 已确认决策

| 决策项 | 结论 |
| --- | --- |
| 中央索引 | `%APPDATA%\QuickRec\recordings.json` |
| 存储形态 | 本地 JSON，不使用 SQLite 或云服务 |
| 索引上限 | 最多 200 条 |
| 默认展示 | 最近 50 条，每次“加载更多”增加 50 条 |
| 超限处理 | 只淘汰最旧索引，不删除视频或旁车文件 |
| 首次自动迁移 | 只迁移当前视频保存路径下的 v1.5 旧索引 |
| 其他旧目录 | 用户主动导入；切换到含旧索引的目录时只提示一次 |
| 旧索引处理 | 原样保留，不重命名、不删除，保证 v1.5 回滚可读 |
| 重新定位 | 单项重新定位 + 目录批量重建 |
| 重建扫描范围 | 只扫描所选目录当前层级的 `QuickRec_*.mp4`，不递归 |
| 去重 | 规范化绝对路径优先，文件大小和修改时间辅助 |
| 疑似匹配 | 仅作为重新定位候选，必须由用户确认 |
| 素材 ID | 保留旧 ID；新记录生成随机稳定 ID；移动后 ID 不变 |
| 元数据 | 时长、分辨率、FPS、模式、音频、文件大小 |
| 旧素材元数据 | 使用打包内 `ffprobe` 按需解析；失败时保留不完整记录 |
| 损坏恢复 | 原子写入、自动恢复最近有效备份、保留损坏原件 |
| 备份上限 | 有效备份 1 份，损坏归档最多 5 份 |
| 导入/重建 | 后台扫描、可取消、预览摘要、确认后一次性写入 |
| 索引写入失败 | 视频仍判定保存成功，独立警告并支持重试入库 |
| 用户入口 | “最近录制”统一升级为“素材库” |
| 详情布局 | 左侧列表 + 右侧详情；窄窗口切换为上下布局 |
| 从素材库移除 | 只移除索引，不删除视频 |
| 删除视频 | 单个 MP4 移入 Windows 回收站，成功后移除索引 |
| 关联文件 | 不删除诊断日志、旧索引、目录或未知旁车文件 |
| 未来兼容 | `schema_version` + 稳定 ID + 通用 `extensions` |
| 原型 | 更新现有 Full HTML 原型，不建立第二套事实源 |
| 工程边界 | 抽取 `RecordingLibraryService`，不全面重写核心类 |
| 质量门禁 | 本轮新增/修改模块执行 ruff、mypy；新增核心模块覆盖率不低于 80% |
| v1.5.1 必修项 | 合并进 v1.6，作为独立 D0 发布债收口模块 |

### 1.4 当前事实与限制

- v1.5 索引位于 `<save_path>\QuickRecMetadata\recordings.json`，默认最多 50 条。
- v1.5 记录已经包含独立 `id`，但缺少分辨率、FPS 和完整时长写入。
- 当前 `RecentRecordingsDialog` 读取当前保存路径，没有跨路径统一视图。
- 当前 PyInstaller 包只明确携带 `ffmpeg.exe`，尚未携带 `ffprobe.exe`。
- 当前 ruff、mypy 和 coverage 均排除部分入口与 UI，v1.6 只扩大本轮相关模块的真实覆盖。
- `%APPDATA%` 仅作为本机索引存放位置；本版不提供跨设备同步语义。

## 2. 背景与问题

### 2.1 产品阶段

QuickRec Full v1.5 已完成从“录制后只得到一个文件”到“产品内可以查看最近录制”的第一步。用户可以打开文件、打开目录、复制路径和移除失效索引，但历史仍绑定当前保存路径。

Full 的长期方向是创作者工作台。要继续进入项目管理、预览、编辑、导出和未来 AI 处理，必须先建立不会随保存路径切换而变化的素材身份和可靠索引。

### 2.2 主要问题

#### 问题 1：历史记录随保存路径分散

用户切换保存路径后，最近录制窗口只读取新目录下的索引。旧视频仍在磁盘中，但产品内视图被割裂，用户需要记住每个目录并手动切换。

#### 问题 2：索引损坏后缺少恢复闭环

v1.5 读取失败时返回错误，不破坏视频，但没有自动备份恢复、损坏原件归档和目录重建。索引一旦损坏，用户会失去对历史记录的信任。

#### 问题 3：文件移动后只能移除，不能重新关联

当前记录会显示“文件已移动或删除”，但用户只能从列表移除。对于只是整理目录或移动文件的情况，素材身份和已有元数据无法保留。

#### 问题 4：素材信息不足

现有记录缺少稳定写入的时长、分辨率和 FPS。多条相似录制并存时，用户仍需要逐个打开才能判断内容是否适用。

#### 问题 5：历史逻辑继续进入核心编排类会扩大回归面

若把迁移、恢复、扫描和删除继续堆进 `QuickRecApp` 或 `RecorderManager`，录制主流程与素材管理会形成高耦合。v1.6 需要一个受控服务边界，但不应借机全面重构。

## 3. 目标

### 3.1 用户目标

- 切换视频保存路径后，仍能在同一个素材库看到历史录制。
- 升级到 v1.6 后，当前目录中的 v1.5 历史可以安全迁移且不会丢失。
- 索引损坏时能够自动恢复，恢复失败时可以通过目录重建找回素材。
- 文件移动后能够重新定位并保留原素材身份。
- 用户能通过时长、分辨率、大小和状态快速识别素材。
- 用户可安全地从素材库移除记录，或明确选择把单个视频移入回收站。

### 3.2 产品目标

- 建立 QuickRec Full 的中央素材索引和稳定素材 ID。
- 将 v1.5 最近录制能力平滑升级为跨路径轻量素材库。
- 为后续完整工作台提供可靠、可迁移、可扩展的数据地基。
- 保持录制保存优先，素材索引失败不得误报录制失败。
- 提高数据迁移和恢复代码的自动化验证可信度。

### 3.3 版本成功标准

v1.6 完成后应满足：

1. 所有新录制统一写入 `%APPDATA%\QuickRec\recordings.json`。
2. 当前保存路径中的 v1.5 历史可幂等迁移，旧索引原样保留。
3. 用户可导入其他旧保存目录，并在确认后合并中央索引。
4. 中央索引损坏时自动恢复最近备份，损坏原件可追溯。
5. 用户可通过目录重建和单项重新定位恢复文件关联。
6. 素材库最多保存 200 条，默认展示 50 条并支持加载更多。
7. 新录制具备完整基础元数据；旧素材可通过 `ffprobe` 补齐。
8. 素材库打开、列表交互和后台扫描达到约定性能口径。
9. D0 发布债、相关自动化检查和打包 GUI 验收全部通过。

## 4. 方案收敛与优先级

### 4.1 候选方案

| 方案 | 内容 | 优点 | 风险 | 结论 |
| --- | --- | --- | --- | --- |
| 方案 A：继续按保存路径维护最近 50 条 | 只补元数据和重新定位 | 改动最小 | 无法解决跨路径割裂和长期演进问题 | 不采用 |
| 方案 B：中央 JSON 轻量素材库 | 集中索引、迁移恢复、重建、稳定 ID | 与现有实现兼容，范围可控，可回滚 | 需要严谨迁移和冲突处理 | 采用 |
| 方案 C：数据库和完整工作台 | SQLite、项目、预览、编辑、导出 | 长期能力完整 | 范围与验证矩阵失控 | 不采用 |

### 4.2 推荐方案

采用 **方案 B：中央 JSON 轻量素材库**。

推荐理由：

- 直接解决跨保存路径历史割裂。
- 可复用 v1.5 的 JSON 模型、稳定 ID 和 UI 入口。
- 不引入数据库、账号和云服务，符合本地轻量边界。
- 支持备份、恢复和回滚，不要求一次性重建工作台。
- 未来可通过 `schema_version` 和 `extensions` 增量演进。

## 5. 范围

### 5.1 D0 发布债收口

- 修复 CI 中开机自启测试对真实 HKCU 的依赖，使用 mock 或隔离方式验证。
- 建立单一版本事实源，诊断、构建和产品文档读取一致版本号。
- 收口 v1.5 发布事实、Git 状态和当前事实入口。
- D0 与素材库功能分开验收，但 D0 未通过时 v1.6 不得发布。

### 5.2 M1 中央素材索引

- 索引固定存放在 `%APPDATA%\QuickRec\recordings.json`。
- 根结构包含 `schema_version`、`max_items`、素材列表、迁移来源状态和可选扩展区。
- 索引最多保留 200 条，按原始录制时间倒序排列。
- 超出上限时只移除最旧索引，不删除磁盘文件。
- 新录制成功后写入中央索引，不再把新记录写入保存目录下的旧索引。
- 索引写入采用临时文件和原子替换。

### 5.3 M2 v1.5 迁移与手动导入

- v1.6 首次运行只检查当前保存路径下的旧索引。
- 自动迁移前读取旧索引、校验结构、执行去重并生成摘要。
- 迁移成功后显示一次性摘要，不重复打扰。
- 用户可从素材库主动选择其他旧保存目录导入。
- 用户切换保存路径时，如目标目录存在未处理旧索引，每个目录只提示一次。
- 旧索引原样保留，中央索引记录来源和处理状态。
- 重复迁移必须幂等，不产生重复素材或新 ID。

### 5.4 M3 备份恢复、重建与重新定位

- 每次覆盖中央索引前保留最近一份有效备份。
- 读取中央索引失败时，将损坏原件按时间戳归档，再尝试恢复有效备份。
- 损坏归档最多保留 5 份，超限只清理最旧归档。
- 恢复失败时显示可处理状态，允许用户进入目录重建。
- 目录重建只扫描用户选择目录当前层级的 `QuickRec_*.mp4`。
- 不递归、不扫描磁盘、不导入其他命名视频。
- 单条缺失记录支持选择新文件重新定位。
- 批量扫描发现疑似匹配时，作为重新定位候选展示，必须由用户确认。

### 5.5 M4 素材库 UI

- 托盘菜单和录制完成结果条中的“最近录制”统一更名为“素材库”。
- 两个入口打开同一个素材库窗口实例或一致窗口状态。
- 默认展示最近 50 条，每次加载更多 50 条，最多展示 200 条。
- 布局为左侧素材列表、右侧详情；窄窗口切换为上下布局。
- 支持打开文件、打开所在目录、复制路径、重新定位、从素材库移除。
- 支持将单个 MP4 移入 Windows 回收站。
- 提供导入旧目录、目录重建和索引恢复状态入口。
- 不提供搜索、筛选、标签、收藏、内嵌预览和批量删除。

### 5.6 M5 素材元数据

- 新录制直接写入录制时已知元数据。
- 历史导入和目录重建使用打包内 `ffprobe` 按需解析。
- 解析失败时仍保留基础记录，字段为空并显示“元数据不完整”。
- 默认列表显示文件名、录制时间、时长、分辨率、文件大小和文件状态。
- 详情显示 FPS、录制模式、音频来源、完整路径、诊断目录和导入来源。

### 5.7 M6 最小服务拆分

- 新增 `RecordingLibraryService` 作为中央素材库业务边界。
- 服务负责索引读写、迁移、导入、备份恢复、目录重建、重新定位和删除结果同步。
- `QuickRecApp` 只负责窗口装配、主线程 UI 回调和录制完成事件转交。
- `RecorderManager` 保持录制生命周期职责，不承担素材库管理。
- 不借机全面重写 `QuickRecApp`、`RecorderManager` 或捕获链路。

### 5.8 M7 原型与工程门禁

- 更新 `doc/prototypes/product-prototype/full.html` 的现有 Full 原型。
- 原型覆盖正常、空、缺失、迁移、导入、重建、恢复、失败、取消和删除确认状态。
- 本轮新增和修改模块纳入 ruff、mypy。
- 新增核心模块测试覆盖率不得低于 80%。
- 与本轮相关的 UI 纯逻辑应从现有排除项中逐步纳入验证。

### 5.9 明确不包含

- SQLite、其他数据库、账号、云同步或跨设备素材库。
- 项目管理、内嵌视频预览、时间线、剪辑和导出队列。
- AI 模型下载、字幕、摘要、章节、标签、实时识别或云端 AI。
- 缩略图批量生成、收藏、搜索、筛选和复杂分页。
- 摄像头采集、摄像头布局和画面编排。
- WGC 正式接入、dxcam/WGC 用户切换、120 FPS 和多显示器正式支持。
- 全量重写 `QuickRecApp`、`RecorderManager`。
- 自动扫描磁盘、递归扫描目录或导入非 QuickRec 视频。
- 永久删除视频、批量删除或自动删除关联文件。
- 修改 QuickRec Lite 的代码、文档或发布范围。

## 6. 用户与场景

### 6.1 目标用户

- 经常切换保存目录的录屏用户。
- 需要回看、定位和复用多条录制结果的创作者。
- 从 v1.5 升级并希望保留历史记录的现有用户。
- 手动整理视频目录后需要恢复素材关联的用户。

### 6.2 核心场景

#### 场景 A：升级后迁移当前历史

用户升级到 v1.6，应用检测当前保存目录中的 v1.5 索引。迁移完成后显示摘要，用户打开素材库即可看到原有记录；旧索引仍留在原目录。

#### 场景 B：切换保存路径但保留统一视图

用户将保存路径从 A 切换到 B。新录制写入 B，但素材库仍同时展示 A 与 B 中的素材。

#### 场景 C：导入其他旧目录

用户在素材库选择“导入旧目录”，扫描结果显示新增、重复、候选和失败数量。用户确认后统一写入中央索引。

#### 场景 D：索引损坏恢复

中央索引损坏后，应用保存损坏副本并自动恢复最近有效备份。若无有效备份，用户可选择目录重建，不影响继续录制。

#### 场景 E：文件移动后重新定位

素材显示缺失。用户选择新的 MP4，确认后更新路径，素材 ID、原始时间和已有元数据保持不变。

#### 场景 F：安全删除视频

用户明确选择“删除视频文件”，确认后 MP4 被移入 Windows 回收站，索引同步移除；诊断目录和未知旁车文件不受影响。

## 7. 完整功能链路

### 7.1 新录制入库

1. 用户启动任一录制模式。
2. 录制核心完成视频编码、音频混合和最终保存。
3. 主流程确认最终 MP4 路径有效。
4. 录制会话提供模式、音频、时长、分辨率和 FPS。
5. `RecordingLibraryService` 构建素材记录并读取文件大小、修改时间。
6. 服务加载中央索引、去重、排序、裁剪并原子写入。
7. 成功时结果条显示录制保存成功，并提供“素材库”入口。
8. 失败时视频仍判定保存成功，额外提示“素材索引写入失败”。
9. 用户可选择“重试加入素材库”或“打开视频目录”。

### 7.2 首次自动迁移

1. v1.6 首次启动读取迁移状态。
2. 只检查当前保存路径的 `QuickRecMetadata\recordings.json`。
3. 不存在旧索引时记录“无可迁移数据”，不弹提示。
4. 存在时后台解析、校验和去重。
5. 解析成功后准备迁移结果，不修改旧索引。
6. 原子写入中央索引。
7. 展示一次性迁移摘要。
8. 写入失败时保留中央索引原状态，并提供重试、打开旧目录和稍后处理。

### 7.3 手动导入与目录重建

1. 用户打开素材库，点击“导入旧目录”或“重建目录”。
2. 用户选择目录；取消选择时关闭流程，不改变索引。
3. 后台扫描开始，UI 显示进度并允许取消。
4. 导入优先读取旧索引；重建只扫描当前层级 `QuickRec_*.mp4`。
5. 缺失元数据通过 `ffprobe` 按需解析。
6. 扫描完成后显示新增、重复、重新定位候选、失败和跳过数量。
7. 用户可检查候选匹配并确认或取消。
8. 确认后服务一次性原子写入中央索引。
9. 成功反馈包含实际写入数量；失败反馈包含可重试动作和日志位置。
10. 用户关闭或取消时不写入部分结果。

### 7.4 素材重新定位

1. 用户选择缺失素材。
2. 点击“重新定位”。
3. 文件选择器默认打开原目录或当前保存目录。
4. 用户选择 MP4；取消时保持原记录。
5. 系统校验文件存在且可读取。
6. 更新路径、目录、文件名、大小和修改时间，保留素材 ID 与原始录制时间。
7. 必要时用 `ffprobe` 补充缺失元数据。
8. 原子保存并反馈成功；失败时保留旧记录和错误上下文。

### 7.5 从素材库移除

1. 用户选择素材并点击“从素材库移除”。
2. UI 明确说明不会删除视频文件。
3. 用户确认后只移除索引记录。
4. 写入失败时记录和视频都保持不变，并显示失败反馈。

### 7.6 删除视频文件

1. 用户选择可用素材并点击“删除视频文件”。
2. 二次确认明确显示文件名、完整路径和“将移入 Windows 回收站”。
3. 用户取消时不执行任何操作。
4. 系统只处理所选 MP4，不处理关联目录或其他文件。
5. 移入回收站成功后移除中央索引记录。
6. 回收站操作失败时保留索引，显示错误并写入诊断日志。
7. 索引更新失败但视频已移入回收站时，将记录标记为缺失并提示用户重试清理索引，不能伪装成完整成功。

## 8. 数据设计

### 8.1 文件位置

```text
%APPDATA%\QuickRec\recordings.json
%APPDATA%\QuickRec\recordings.json.bak
%APPDATA%\QuickRec\recordings.corrupt-YYYYMMDD-HHMMSS.json
```

该索引属于本机应用数据。即使 `%APPDATA%` 位于 Roaming 路径，本版也不承诺跨设备复制后路径仍有效。

### 8.2 根结构

```json
{
  "schema_version": 2,
  "max_items": 200,
  "items": [],
  "migration_sources": [],
  "extensions": {}
}
```

| 字段 | 类型 | 必需 | 说明 |
| --- | --- | --- | --- |
| `schema_version` | integer | 是 | 数据结构版本，升级必须显式迁移 |
| `max_items` | integer | 是 | 当前上限，v1.6 固定为 200 |
| `items` | array | 是 | 素材记录，按录制时间倒序 |
| `migration_sources` | array | 是 | 旧索引来源、提示和迁移状态 |
| `extensions` | object | 否 | 通用扩展区；v1.6 不写 AI 业务字段 |

### 8.3 素材记录

| 字段 | 类型 | 必需 | 说明 |
| --- | --- | --- | --- |
| `id` | string | 是 | 稳定素材 ID；迁移保留旧值 |
| `file_path` | string | 是 | 当前绝对路径 |
| `normalized_path` | string | 是 | Windows 规范化路径，用于快速去重 |
| `file_name` | string | 是 | 文件名 |
| `directory` | string | 是 | 当前目录 |
| `mode` | string | 是 | `fullscreen`、`area`、`window` 或 `unknown` |
| `audio_source` | string | 是 | `none`、`system`、`microphone`、`both` 或 `unknown` |
| `created_at` | string | 是 | 原始录制时间，ISO 8601 |
| `imported_at` | string/null | 否 | 导入中央索引时间，不参与默认排序 |
| `duration_sec` | number/null | 否 | 视频时长 |
| `width` | integer/null | 否 | 输出宽度 |
| `height` | integer/null | 否 | 输出高度 |
| `fps` | number/null | 否 | 输出帧率 |
| `file_size_bytes` | integer/null | 否 | 文件大小 |
| `file_modified_ns` | integer/null | 否 | 文件修改时间辅助指纹 |
| `status` | string | 是 | `available`、`missing`、`metadata_incomplete` |
| `diagnostic_dir` | string/null | 否 | 录制时诊断目录 |
| `source_type` | string | 是 | `recording`、`migration`、`import`、`rebuild` |
| `source_history_path` | string/null | 否 | 迁移/导入来源旧索引路径 |
| `extensions` | object | 否 | 后续兼容，不预定义 AI 内容 |

### 8.4 素材 ID 规则

- v1.5 记录存在 ID 时原样保留。
- 新录制、缺 ID 的旧记录和目录重建记录生成随机稳定 ID。
- 文件改名、移动和重新定位不得改变 ID。
- 路径和文件指纹不得替代 ID。
- 去重合并时保留已存在记录 ID，不为同一素材生成新 ID。

### 8.5 去重与匹配规则

1. 规范化绝对路径相同：视为同一素材。
2. 已有记录缺失，扫描文件的文件名、大小和修改时间均相符：标记为重新定位候选。
3. 辅助指纹不完全相符：作为新素材或冲突项，不自动覆盖。
4. v1.6 不计算完整视频 SHA256。
5. Windows 路径比较必须处理大小写、分隔符和冗余相对段。

### 8.6 时间排序规则

1. 旧索引存在 `created_at` 时保留原值。
2. 目录重建优先解析 QuickRec 文件名中的录制时间。
3. 无法解析时使用文件修改时间。
4. `imported_at` 只用于审计，不参与默认排序。

### 8.7 备份与恢复规则

- 写入新索引前，确认当前索引可解析后再覆盖有效备份。
- 临时文件必须与目标文件位于同一目录，写入完成后原子替换。
- 损坏文件必须先归档，再执行恢复，禁止直接覆盖。
- 自动恢复成功后仍保留本次损坏归档，并在 UI 和日志中反馈。
- 自动恢复失败时不得创建空索引覆盖损坏数据。
- 用户明确确认目录重建结果后，才允许创建新的有效中央索引。

### 8.8 迁移状态

每个旧目录至少记录：

- 规范化来源索引路径；
- 首次检测时间；
- 是否已提示；
- 用户选择：导入、稍后、拒绝；
- 最近迁移时间与结果数量；
- 最近错误摘要。

同一来源迁移重试不得重复生成素材。

## 9. 前端与交互设计

### 9.1 原型输出模式

更新现有文件：

```text
doc/prototypes/product-prototype/full.html
doc/prototypes/product-prototype/prototype-design.md
```

原型必须覆盖：

- 素材库入口与默认列表；
- 左列表、右详情及窄窗口布局；
- 加载更多；
- 首次迁移摘要；
- 手动导入和目录重建进度；
- 扫描预览与候选确认；
- 空状态、缺失状态、元数据不完整；
- 自动恢复成功和恢复失败；
- 索引写入失败；
- 从素材库移除确认；
- 移入回收站确认与失败反馈。

### 9.2 主窗口原型

```text
┌────────────────────── QuickRec 素材库 ──────────────────────┐
│ 素材库                                      [导入旧目录] [重建] │
│ 已收录 126 条 · 默认显示最近 50 条                           │
├──────────────────────────┬───────────────────────────────────┤
│ 录制列表                  │ 素材详情                          │
│                           │                                   │
│ QuickRec_2026...mp4       │ QuickRec_2026...mp4               │
│ 07-10 14:30 · 00:12:18    │ 1920×1080 · 60 FPS · 245 MB       │
│ 1920×1080 · 245 MB · 可用 │ 窗口录制 · 系统声音               │
│                           │ E:\Videos\...                      │
│ QuickRec_2026...mp4       │                                   │
│ 文件已移动或删除          │ [打开] [打开目录] [复制路径]       │

├──────────────────────────┴───────────────────────────────────┤
│ 状态与反馈                                                   │
└──────────────────────────────────────────────────────────────┘
```

### 9.3 列表状态

| 状态 | 展示 | 可用操作 |
| --- | --- | --- |
| 可用 | 正常元数据和“可用”状态 | 打开、目录、复制、移除、回收站删除 |
| 缺失 | “文件已移动或删除” | 重新定位、复制旧路径、移除 |
| 元数据不完整 | 显示已知字段，未知字段为 `-` | 正常打开、重新解析元数据 |
| 空 | “暂无素材”及录制/导入引导 | 导入旧目录、重建目录 |
| 加载失败 | 明确错误与恢复状态 | 重试、恢复备份、重建目录、打开日志目录 |

### 9.4 迁移摘要

推荐文案：

```text
历史录制已迁移
新增 37 条，跳过重复 2 条，元数据不完整 1 条。

[查看素材库] [打开旧目录] [关闭]
```

失败时：

```text
历史录制迁移未完成
旧索引和视频文件均未修改，可稍后重试。

[重试] [打开旧目录] [稍后处理]
```

### 9.5 后台扫描与取消

- 扫描运行时主窗口仍可移动和关闭确认弹窗。
- 用户关闭素材库时若扫描仍在进行，应询问是否取消扫描。
- 取消请求发出后 2 秒内停止继续调度新解析任务。
- 已读取的临时结果随流程结束释放，不写入中央索引。
- 扫描完成后必须先展示摘要，不能自动提交。

### 9.6 删除文案

从素材库移除：

```text
从素材库移除这条记录？
视频文件不会被删除，之后可通过目录重建再次加入。
```

删除视频：

```text
将此视频移入 Windows 回收站？
仅处理所选 MP4，不会删除诊断日志或其他文件。
```

两个操作必须使用不同按钮名称、说明和确认内容，不能共享模糊的“删除”文案。

### 9.7 视觉与可用性要求

- 延续当前 QuickRec Full 安静、工具型界面，不制作营销式页面。
- 素材库是可调整大小的工作窗口，不使用嵌套卡片。
- 常用文件操作使用清晰图标加工具提示；危险操作不得与打开操作相邻且同权重。
- 长路径使用省略显示，但复制结果必须是完整路径。
- 所有中文文案在 100%、150%、200% DPI 下不得重叠或截断。
- 后台任务必须有明确进度、取消和完成反馈。

## 10. 详细功能需求

### 10.1 中央索引解析

- 首次无文件时返回空素材库，不视为错误。
- 支持 UTF-8 与 UTF-8 BOM 读取。
- 根节点、版本和列表类型异常时进入损坏恢复流程。
- 单条记录字段异常时尽量降级读取，并记录跳过原因；不得因一条坏记录丢弃全部有效记录。
- 未知 `extensions` 内容必须原样保留，不能在普通写入时丢失。

### 10.2 新录制写入

- 只在最终 MP4 保存成功后入库。
- 记录写入失败不得改变视频成功状态。
- 同一保存回调重复触发时不得产生重复记录。
- 写入成功后素材库已打开时应刷新列表，但不得抢占用户当前选择。
- 当前列表只加载 50 条时，新录制插入顶部，并保持加载计数正确。

### 10.3 迁移

- 迁移 v1.5 `schema_version=1` 数据到 v1.6 当前版本。
- 保留已有 ID、路径、创建时间、模式、音频、时长、大小和诊断目录。
- 补充 `normalized_path`、来源、导入时间和缺失字段。
- 超过 200 条时按原始录制时间裁剪，摘要必须说明淘汰数量。
- 迁移失败不写半成品，不修改旧索引。

### 10.4 手动导入

- 入口名称为“导入旧目录”。
- 选择目录后优先查找 `QuickRecMetadata\recordings.json`。
- 旧索引不存在或无法读取时，询问是否改用目录重建。
- 同一目录已成功迁移时允许再次检查，但结果应显示“无新增记录”。
- 用户拒绝切换路径提示后，可随时从素材库手动导入。

### 10.5 目录重建

- 只匹配当前层级 `QuickRec_*.mp4`，扩展名比较不区分大小写。
- 单次最多接受 200 个候选；超过时按推断录制时间选择最近 200 个并提示。
- 无法读取的视频进入失败清单，不阻断其他素材扫描。
- 解析元数据期间不得长时间占用 Qt 主线程。
- 用户确认前不得改变中央索引。

### 10.6 重新定位

- 只允许选择存在且可读的 MP4。
- 选择已经被其他素材使用的路径时，应提示重复并定位已有素材，不覆盖当前记录。
- 用户确认候选匹配后保留旧 ID。
- 重新定位成功后状态更新为可用。
- 更新失败时保持原缺失记录。

### 10.7 加载更多

- 首次加载 50 条。
- 每次增加 50 条，直至索引末尾或 200 条。
- 没有更多数据时隐藏或禁用按钮，并显示总数。
- 加载更多不得重新读取或解析全部视频元数据。

### 10.8 删除与回收站

- “从素材库移除”只修改中央索引。
- “删除视频文件”只对状态可用的单条素材开放。
- 删除必须调用 Windows 回收站能力，不允许 `unlink` 或永久删除作为回退。
- 回收站能力不可用时操作失败，不得改用永久删除。
- 视频进入回收站后索引更新失败时，记录必须在下次加载被识别为缺失。

### 10.9 设置与入口

- 中央索引路径不提供用户配置项。
- 视频保存路径设置保持现有行为。
- 切换保存路径后的旧索引提示不得阻塞设置保存。
- 托盘空闲和录制中菜单均保留素材库入口。
- 录制结果条提供“素材库”入口，不增加第二个“最近录制”入口。

## 11. 影响范围

### 11.1 前端 / UI

预计影响：

- `src/ui/recent_recordings_dialog.py`：升级或替换为素材库窗口。
- `src/ui/tray_icon.py`：菜单文案和回调统一为素材库。
- `src/ui/toolbar.py`：录制结果入口文案更新。
- `src/ui/settings_dialog.py`：保存路径切换后的旧索引提示衔接。
- `doc/prototypes/product-prototype/full.html`：同步高保真原型。

### 11.2 后端 / 录制核心

- `src/main.py`：只接收录制保存事件并调用素材服务。
- `src/recorder/recorder_manager.py`：暴露录制会话已知元数据，不承担索引逻辑。
- 新增素材库服务和数据模型模块。
- 不修改 dxcam 默认捕获后端和 FFmpeg 编码主链路。

### 11.3 日志系统

至少记录以下事件：

- `library load succeeded/failed`
- `library save succeeded/failed`
- `library backup restored`
- `library corrupt file archived`
- `history migration started/completed/failed`
- `history import started/cancelled/completed/failed`
- `library rebuild started/cancelled/completed/failed`
- `recording relink succeeded/failed`
- `recording recycle succeeded/failed`
- `metadata probe failed`

日志不得记录视频内容、音频内容或任何 AI 数据。

### 11.4 配置

- 不新增中央索引路径配置。
- 可新增内部迁移提示状态，但应优先存入中央索引的迁移来源信息。
- 现有保存路径和诊断目录配置保持兼容。
- 统一版本事实源属于 D0 范围。

### 11.5 打包发布

- PyInstaller 产物必须同时携带 `ffmpeg.exe` 和 `ffprobe.exe`。
- 打包测试必须验证 `ffprobe` 可执行、路径解析正确且可读取真实 MP4。
- 若回收站功能引入第三方依赖，必须进入 `requirements`、spec 和许可证审计。
- 不允许开发环境可用但打包产物缺失 `ffprobe` 或回收站依赖。
- 记录 v1.6 发布包大小变化，但体积不是单独阻断项；缺失运行依赖是阻断项。

### 11.6 安全与隐私

- 不自动扫描用户磁盘或未选择目录。
- 不上传路径、元数据、视频或索引。
- 不执行永久删除。
- 导入和重建只在用户明确选择范围内运行。
- 索引包含本地绝对路径，诊断导出沿用当前“本地保存、无需脱敏”的产品约定。

### 11.7 QuickRec Lite

- QuickRec Lite 不实现中央素材库。
- 不修改 `E:\codex\QuickRec-Lite` 的代码、文档、分支或发布产物。
- Full 的 v1.6 文档不得把 Lite 写成当前开发范围。

## 12. 性能与可靠性要求

### 12.1 素材库交互

- 200 条索引下首次打开素材库窗口不超过 1 秒。
- 列表滚动、选择详情和加载更多无明显卡顿。
- 窗口打开不得同步执行全部文件 `ffprobe`。
- 文件状态刷新应控制磁盘访问，不得在每次绘制单元格时重复访问文件系统。

### 12.2 后台任务

- 导入、重建和批量 `ffprobe` 在后台执行。
- Qt 主线程不得因单个视频解析长时间无响应。
- 取消请求 2 秒内停止继续调度新任务。
- 已启动的外部进程应安全结束或回收，不留下后台僵尸进程。
- 扫描总耗时只记录测试基线，不设置脱离样本数量和硬件的固定总时限。

### 12.3 数据可靠性

- 原子写入失败时旧有效索引保持可读。
- 自动恢复不得覆盖损坏原件。
- 迁移、导入和重建取消后中央索引字节内容保持不变。
- 相同来源重复迁移不增加记录数量。
- 视频保存成功率不得因索引服务异常下降。

### 13.5 不通过处理

- 迁移覆盖旧索引、丢失视频或产生不可恢复重复：阻断发布，回滚中央索引写入和迁移入口。
- 索引失败导致录制误报失败：阻断发布，优先恢复录制保存语义。
- 取消后写入部分结果：阻断发布，修复事务边界。
- 回收站失败后执行永久删除：严重阻断，立即移除该删除实现。
- 打包产物缺少 `ffprobe`：阻断发布，不接受开发环境通过代替。
- UI 卡死或后台任务无法取消：进入 bugfix 后重新验收。
- 仅原型问题不影响运行时功能时，记录 bugfix 并修复原型后复验。

## 16. 风险与回滚

### 16.1 风险

| 风险 | 影响 | 控制方式 |
| --- | --- | --- |
| 中央索引损坏 | 素材库不可用 | 原子写入、有效备份、损坏归档、目录重建 |
| 迁移重复或丢记录 | 用户不信任升级 | 稳定 ID、规范路径、幂等测试、旧索引不动 |
| `ffprobe` 扫描阻塞 UI | 素材库卡顿 | 后台执行、按需解析、取消与进度 |
| 辅助指纹误匹配 | 错误重新定位 | 只生成候选，必须人工确认 |
| 回收站操作失败 | 删除状态不一致 | 失败不永久删除，索引按真实文件状态处理 |
| UI 扩大成完整工作台 | v1.6 失控 | 不做预览、项目、编辑、搜索和导出 |
| 服务拆分扩大 | 核心回归 | 只抽素材库职责，录制行为保持不变 |
| 打包体积增加 | 发布包变大 | 记录变化，不以删除运行依赖换体积 |

### 16.2 代码回滚

- 回滚到 v1.5 发布点时，原保存目录中的 v1.5 旧索引仍可使用。
- v1.6 中央索引是附加数据，v1.5 不读取它，不影响旧版录制。
- 回滚不得删除 `%APPDATA%\QuickRec\recordings.json`，以便重新升级。
- 若素材库 UI 阻断发布，可回退入口和服务接入，但不得留下半迁移写入逻辑。

### 16.3 数据回滚

- 旧索引不修改，因此不需要反向迁移。
- 中央索引可保留或人工归档，不能在卸载/回滚时自动删除。
- 视频文件始终是事实源；素材索引不得成为视频可用性的单点依赖。

## 17. 后续路线兼容

### 17.1 完整工作台

v1.6 只提供稳定素材 ID、中央索引、元数据和恢复能力。后续项目管理、预览、基础编辑和导出队列必须在独立版本中定义，不得在 v1.6 实现过程中顺带加入。
