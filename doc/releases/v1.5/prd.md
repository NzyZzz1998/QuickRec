> 类型：正式推进型 PRD
> 状态：已实现并验收
> 产品线：QuickRec Full
> 原型：`doc/prototypes/product-prototype/full.html`

## 1. 需求状态

### 1.1 PRD 类型

本需求为 **正式推进型 PRD**。

判断依据：

- v1.4 已完成稳定性与工程化基线。
- v1.4.1 已完成诊断导出能力。
- 用户已确认 v1.5 以 QuickRec Full 的未来演进目标为主线。
- 用户已确认 v1.5 版本定位为“捕获体验与创作者工作台地基版”。
- 用户已确认自绘光标从录制输出链路中移除，鼠标点击高亮保留。
- 用户已确认 Full 工作台第一步选择“录制历史 / 素材入库”。
- 用户已确认 WGC / 新捕获后端仅作为技术研究 spike，不替换默认捕获链路。

### 1.2 已确认决策

| 决策项 | 结论 |
| --- | --- |
| 版本名称 | QuickRec Full v1.5 捕获体验与创作者工作台地基版 |
| 产品线 | 只面向 QuickRec Full，不改变 QuickRec Lite 范围 |
| 自绘光标 | 全屏、区域、窗口录制均不再主动把自绘光标叠加到视频帧 |
| 鼠标点击高亮 | 保留，仍作为屏幕叠加层，不写入视频帧 |
| 工作台第一步 | 录制历史 / 素材入库 |
| 最近记录数量 | 默认保留最近 50 条 |
| 历史入口 | 托盘菜单 + 录制完成结果条均提供入口 |
| 素材记录存储 | 默认在视频保存目录内新建独立文件夹 |
| 文件丢失处理 | 提供“从列表移除”操作 |
| 原型 | 需要同步更新 Full 原型 |
| WGC / 新捕获后端 | v1.5 仅做技术研究 spike，不替换默认 dxcam 链路 |

### 1.3 当前假设

- v1.5 仍保留 Full 现有三种录制模式：全屏、区域、窗口。
- v1.5 不降低 v1.4.1 诊断导出能力。
- v1.5 不改 Lite 仓库和 Lite 发布范围。
- v1.5 的录制历史 / 素材入库使用本地 JSON 文件，不引入 SQLite、账号、云同步或数据库服务。
- v1.5 的“素材入库”先服务“找到、识别、打开、复用录制结果”，不进入复杂剪辑。

## 2. 背景与问题

### 2.1 当前阶段

QuickRec Full 已完成：

- v1.4 稳定性与工程化基线。
- v1.4.1 诊断导出能力。
- Full / Lite 工作区拆分。
- Full 当前发布点和文档事实源治理。

历史文档中已经明确 Full 的长期方向是创作者工作台，面向录制、管理、轻编辑、导出和质量诊断的完整工作流。但 Full Workbench 原型也明确它不属于 v1.4 强制开发范围。

因此 v1.5 的合理定位不是一次性做完整工作台，而是从现有录制工具向 Full 工作台迈出第一步：先解决捕获观感风险，再把录制结果从“散落在保存目录里的文件”提升为“可被 Full 继续管理的素材”。

### 2.2 主要问题

#### 问题 1：自绘光标链路影响录制观感和维护信任

历史测试中出现过窗口录制光标比例异常、大鼠标闪现、DPI 误差等问题。v1.4 已对窗口录制采取“不叠加鼠标光标”的策略，但当前 Full 代码中全屏 / 区域仍存在自绘光标叠加链路。

这会带来两个风险：

- 用户回放时看到的光标可能与系统真实光标不一致。
- 后续继续修自绘光标会消耗维护成本，但不一定能达到原生捕获质量。

#### 问题 2：录制结果缺少产品内管理入口

当前录制结果主要依赖保存目录、Toast、结果条和系统资源管理器。对于 Full 方向来说，这还停留在“单次录制工具”，没有形成后续素材管理、轻编辑、导出队列和质量诊断的数据地基。

#### 问题 3：新捕获后端有潜在价值，但不适合直接替换主链路

WGC 等系统级捕获后端可能改善窗口捕获、光标原生捕获和现代 Windows 兼容性，但当前仍缺少实际 spike 证据。直接替换 dxcam 会放大回归风险。

v1.5 只把它作为研究项，回答“是否值得后续进入正式 PRD”，不在本版替换默认捕获链路。

## 3. 目标

### 3.1 用户目标

- 用户录制后能在应用内快速看到最近录制内容，不必只依赖系统文件夹。
- 用户能快速打开录制结果、定位文件、复制路径。
- 用户不会再因为自绘光标异常看到明显不自然的鼠标贴图。
- 用户在录制文件被移动或删除后，能看到明确提示，并可从列表移除无效记录。

### 3.2 产品目标

- 为 QuickRec Full 从单次录制工具演进到创作者工作台建立第一层数据和入口。
- 降低自绘光标相关的视觉质量风险和维护成本。
- 保持当前稳定录制链路不回退。
- 为后续捕获后端升级提供技术 spike 材料。

### 3.3 版本目标

v1.5 完成后，QuickRec Full 应具备：

1. 录制输出不再依赖自绘光标叠加。
2. 鼠标点击高亮仍可作为屏幕叠加层使用，但不写入视频帧。
3. 录制完成后生成可管理的历史 /素材记录。
4. 用户可通过托盘菜单和录制完成结果条查看最近录制。
5. 最近录制默认保留 50 条，旧记录自动裁剪。
6. 用户可打开文件、打开目录、复制路径、从列表移除无效记录。
7. WGC / 新捕获后端完成独立研究记录，不影响默认 dxcam 链路。

## 4. 方案收敛与优先级

### 4.1 候选方案

| 方案 | 内容 | 优点 | 缺点 | 结论 |
| --- | --- | --- | --- | --- |
| 方案 A：只移除自绘光标 | 删除 / 禁用自绘光标叠加，保留现有保存方式 | 成本低，风险小 | 不能推进 Full 工作台方向 | 不采用 |
| 方案 B：自绘光标收口 + 最近录制 / 素材入库 | 解决光标观感风险，并增加最小素材管理入口 | 与 Full 方向一致，范围可控 | 需要新增本地记录模型和 UI 入口 | 采用 |
| 方案 C：完整 Full Workbench | 一次性做项目、素材、轻编辑、导出、质量中心 | 方向完整 | 范围过大，开发和验收风险高 | 不采用 |

### 4.2 推荐方案

采用 **方案 B：自绘光标收口 + 最近录制 / 素材入库**。

推荐理由：

- 自绘光标是明确的体验风险，适合 v1.5 先收口。
- 最近录制 / 素材入库是 Full 工作台的最小地基。
- 不直接做完整工作台，可以保持个人项目迭代范围可控。
- 不替换 dxcam 默认链路，能保护 v1.4.1 已验证稳定性。

### 4.3 优先级策略

```text
优先级 = Full 演进目标匹配度 + 用户高频路径价值 + 风险降低
       - 开发复杂度 - 对现有录制链路的回归风险 - 后续维护成本
```

| 需求 | 优先级 | 进入 v1.5 | 理由 |
| --- | --- | --- | --- |
| 移除录制输出中的自绘光标叠加 | P0 | 是 | 已有历史风险，影响录制观感和信任 |
| 保留鼠标点击高亮但不写入视频帧 | P0 | 是 | 用户已确认保留，且历史设计为屏幕叠加层 |
| 录制历史 / 素材入库最小闭环 | P0 | 是 | Full 工作台第一步，承接现有录制结果 |
| 最近录制入口 | P1 | 是 | 提升录制后找文件效率 |
| 素材基础元数据 | P1 | 是 | 为后续剪辑、导出、诊断做地基 |
| 文件丢失和从列表移除 | P1 | 是 | 避免历史列表失真 |
| WGC / 新捕获后端 spike | P2 | 是 | 只做研究，不影响主链路 |
| 完整轻剪辑 | P3 | 延后 | 成本高，依赖素材模型稳定 |
| 导出队列 | P3 | 延后 | 依赖素材/项目模型，不适合作为 v1.5 首发 |
| 模板与预设 | P3 | 延后 | 非当前核心问题 |
| 复杂质量诊断中心 | P3 | 延后 | v1.4.1 已有诊断导出，v1.5 不扩成中心 |

## 5. 范围

### 5.1 本次包含

#### M1 自绘光标收口

- 停止在全屏、区域、窗口录制输出帧中叠加自绘光标。
- 移除或旁路 `cursor_overlay` 在录制输出链路中的调用。
- 保留必要历史测试或改写测试，确保不再绘制自绘光标。
- 保留鼠标点击高亮设置和运行逻辑。
- 确认点击高亮仍是屏幕叠加层，不进入视频帧。

#### M2 录制历史 / 素材入库

- 录制成功后生成一条素材记录。
- 最近录制默认保留 50 条。
- 元数据默认存储在视频保存目录下的独立文件夹：

```text
<save_path>\QuickRecMetadata\recordings.json
```

- 如果保存路径变更，新的录制记录写入新保存路径下的 `QuickRecMetadata`。
- 历史列表读取当前保存路径下的 `QuickRecMetadata\recordings.json`。
- 素材记录写入失败不影响录制保存，只写日志和诊断上下文。

#### M3 最近录制 UI

- 在托盘菜单新增“最近录制”入口。
- 在录制完成结果条新增“查看最近录制”入口。
- 新增轻量“最近录制”窗口或弹窗。
- 最近录制窗口支持打开文件、打开目录、复制路径、从列表移除。
- 文件不存在时显示明确状态，不崩溃。
- 空状态显示“暂无录制记录”。

#### M4 WGC / 新捕获后端 spike

- 输出捕获后端研究记录。
- 重点验证：
  - 是否能原生捕获光标。
  - 是否能改善窗口录制兼容性。
  - 是否能与 PyInstaller 打包兼容。
  - 是否可能与现有 dxcam 链路并存。
- 不替换默认录制链路。
- 不新增用户可见后端切换开关。

#### M5 Full 原型同步

- 更新 `doc/prototypes/product-prototype/full.html`。
- 原型中体现“最近录制 / 素材入库”作为 Full 工作台地基。
- 原型只模拟交互，不调用真实文件系统。
- 更新 `doc/prototypes/product-prototype/prototype-design.md` 中的 v1.5 对应关系。

### 5.2 本次不包含

- 不做完整 Full Workbench。
- 不做项目文件、项目库、复杂素材库。
- 不做视频剪辑、裁剪、拼接、字幕、转场、调色。
- 不做导出队列真实实现。
- 不做模板与预设真实实现。
- 不做复杂质量诊断中心。
- 不做云同步、账号、多设备同步。
- 不改变 Lite 仓库或 Lite v0 范围。
- 不把 WGC 作为默认捕获后端。
- 不开放捕获后端用户切换项。
- 不新增数据库服务或 SQLite。

v1.5 完成后，用户能在不依赖自绘光标的前提下完成稳定录制，并能在应用内查看最近 50 条录制结果、打开文件、打开目录、复制路径、移除无效记录。

| 编号 | 场景 | 前置条件 | 操作 | 预期结果 |
| --- | --- | --- | --- | --- |
| V15-A1 | 全屏录制 | 应用已启动 | 全屏录制 3 秒并播放 | MP4 可播放，不写入自绘光标 |
| V15-A2 | 区域录制 | 应用已启动 | 区域录制 3 秒并播放 | MP4 可播放，不写入自绘光标 |

### 6.3 日志 / 人工检查

需要日志检查：

- `recording history saved`
- `recording history save failed`
- `recording history loaded`
- `recording history pruned`
- `recording history item removed`
- `recording file missing`
- `open recording file failed`
- `open recording directory failed`
- 自绘光标链路禁用或不再调用的测试证据。

需要人工检查：

- 录制视频中不存在异常自绘光标贴图。
- 鼠标点击高亮屏幕叠加层仍可用。
- 最近录制列表信息准确。
- 文件被移动后提示清楚。
- 原型页面能打开且无明显布局错误。

### 6.4 通过标准

- 自动化测试覆盖历史记录写入、读取、保留 50 条、文件不存在、从列表移除、路径复制、录制完成回调。
- 自动化测试证明 `RecorderManager` 不再调用自绘光标叠加。
- 手动验证覆盖全屏、区域、窗口三种模式。
- 手动验证覆盖无声、系统声、麦克风、双音频中至少三类。
- 诊断导出回归通过。
- 原型文件可在浏览器打开并切换到最近录制相关页面。
- v1.5 不引入新的发布包启动失败、无法录制、无法保存问题。

### 6.5 不通过处理

- 如果移除自绘光标导致录制帧处理失败：阻断发布，回退该改动。
- 如果素材记录失败影响录制保存：阻断发布，素材记录必须改为非阻塞。
- 如果最近录制 UI 崩溃：阻断发布。
- 如果历史记录写入失败但视频保存正常：允许继续测试，但必须修复日志和提示。
- 如果 WGC spike 影响默认链路：立即回滚 spike 代码或改为纯文档研究。

## 7. 影响范围

### 7.1 前端 / UI

影响：

- 托盘菜单新增“最近录制”。
- 录制完成结果条新增“查看最近录制”。
- 新增最近录制窗口或弹窗。
- 最近录制列表新增操作按钮：打开、打开目录、复制路径、从列表移除。
- 最近录制列表新增状态：正常、文件不存在、空状态、加载失败。

不影响：

- 设置页中的鼠标点击高亮开关保留。
- 诊断分组保留。
- 现有全屏、区域、窗口录制入口保留。

### 7.2 前端状态

新增状态：

- 历史记录加载中。
- 历史记录为空。
- 历史记录加载失败。
- 文件存在。
- 文件不存在。
- 路径复制成功 / 失败。
- 从列表移除成功 / 失败。

### 7.3 后端 / 录制核心

影响：

- `RecorderManager` 或录制完成回调需要提供录制结果上下文。
- 帧处理链路不再调用自绘光标叠加。
- 录制完成后触发历史记录写入。

约束：

- 历史记录写入不能阻塞录制保存。
- 历史记录写入失败不能改变录制成功结果。
- 录制核心不直接承担 UI 展示逻辑。

### 7.4 数据与文件

新增文件：

```text
<save_path>\QuickRecMetadata\recordings.json
```

推荐 JSON 结构：

```json
{
  "schema_version": 1,
  "max_items": 50,
  "items": [
    {
      "id": "20260710_153022_fullscreen",
      "file_path": "E:\\QRtest\\QuickRec_20260710_153022.mp4",
      "file_name": "QuickRec_20260710_153022.mp4",
      "directory": "E:\\QRtest",
      "mode": "fullscreen",
      "audio_source": "both",
      "created_at": "2026-07-10T15:30:22+08:00",
      "duration_sec": 3.2,
      "file_size_bytes": 18400000,
      "status": "available",
      "diagnostic_dir": "E:\\QRtest\\QuickRecDiagnostics"
    }
  ]
}
```

字段说明：

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `schema_version` | number | 是 | 元数据格式版本，v1.5 固定为 1 |
| `max_items` | number | 是 | 默认 50 |
| `items` | array | 是 | 最近录制记录 |
| `id` | string | 是 | 本地唯一 ID，可由时间戳和模式生成 |
| `file_path` | string | 是 | 录制文件绝对路径 |
| `file_name` | string | 是 | 文件名 |
| `directory` | string | 是 | 文件所在目录 |
| `mode` | string | 是 | `fullscreen` / `region` / `window` |
| `audio_source` | string | 是 | `none` / `system` / `mic` / `both` |
| `created_at` | string | 是 | ISO 时间 |
| `duration_sec` | number/null | 否 | 时长，无法获取时为 null |
| `file_size_bytes` | number/null | 否 | 文件大小，无法获取时为 null |
| `status` | string | 是 | `available` / `missing` |
| `diagnostic_dir` | string/null | 否 | 关联诊断目录 |

兼容策略：

- 文件不存在时视为空历史，不报错。
- JSON 解析失败时记录日志，并显示加载失败；不得影响录制。
- 写入前先读取旧记录，插入新记录后按 `created_at` 倒序保留 50 条。
- 写入建议采用临时文件 + 原子替换，避免异常中断导致 JSON 半写入。

## 9. 用户场景

### 9.1 录制完成后快速复用文件

用户完成一次窗口录制后，在结果条点击“查看最近录制”，列表中出现新记录。用户点击“打开目录”，直接定位到 MP4 所在文件夹。

### 9.2 用户整理文件后发现历史记录失效

用户手动移动了某个 MP4。再次打开最近录制时，该条显示“文件已移动或删除”。用户点击“从列表移除”，该条从历史记录中删除。

### 9.3 用户需要把路径发给别人

用户打开最近录制，点击“复制路径”，将本地路径粘贴到聊天窗口或 issue 记录。

### 9.4 用户开启鼠标点击高亮

用户在设置中开启鼠标点击高亮。录制时屏幕上仍出现点击扩散动画，但输出 MP4 不叠加自绘光标或高亮动画。

### 9.5 开发者评估后续捕获后端

维护者在独立 spike 中验证 WGC 是否能原生捕获光标和窗口内容，输出研究记录。该研究不影响 v1.5 默认录制链路。

## 10. 功能需求

### 10.1 自绘光标收口

- 需求类型：后端 / 录制核心 / 测试。
- 触发条件：任何录制模式写入视频帧前。
- 入口：录制帧处理链路。
- 主流程：
  1. 捕获帧。
  2. 按录制模式进行必要 resize。
  3. 直接写入编码器。
  4. 不调用自绘光标叠加函数。
- 分支流程：
  - 全屏：不叠加自绘光标。
  - 区域：不叠加自绘光标。
  - 窗口：保持当前不叠加自绘光标策略。
- 异常流程：
  - 如果移除后出现帧尺寸或格式异常，按原录制失败路径处理并写日志。
- 状态变化：
  - 不新增用户状态。
- 数据来源：
  - 不新增数据。
- 实现约束：
  - 保留鼠标点击高亮屏幕叠加层。
  - 不删除与区域选择光标、窗口边框高亮相关的 UI 能力。

### 10.2 历史记录服务

- 需求类型：后端 / 数据 / 文件。
- 触发条件：录制成功保存后。
- 入口：录制保存完成回调。
- 主流程：
  1. 获取最终文件路径。
  2. 读取录制模式、音频源、创建时间、文件大小、诊断目录。
  3. 生成素材记录。
  4. 读取 `<save_path>\QuickRecMetadata\recordings.json`。
  5. 插入新记录到列表头部。
  6. 按创建时间倒序保留最近 50 条。
  7. 写入 JSON。
- 分支流程：
  - JSON 文件不存在：创建新文件。
  - `QuickRecMetadata` 不存在：自动创建目录。
  - 旧记录超过 50 条：裁剪旧记录。
- 异常流程：
  - 目录创建失败：写日志，不影响录制保存。
  - JSON 解析失败：写日志，UI 显示加载失败。
  - JSON 写入失败：写日志，不影响录制保存。
- 状态变化：
  - 新增或更新最近录制记录。
- 数据来源：
  - 录制完成回调、配置中的保存路径、文件系统状态。
- 实现约束：
  - 写历史记录必须非阻塞。
  - 不扫描保存目录中的所有 MP4，只记录 QuickRec 自己保存成功后的结果。

### 10.3 最近录制窗口

- 需求类型：前端 / UI / 文件。
- 触发条件：用户点击“最近录制”。
- 入口：
  - 托盘菜单。
  - 录制完成结果条。
- 主流程：
  1. 打开最近录制窗口。
  2. 读取当前保存路径下的历史 JSON。
  3. 展示最近 50 条记录。
  4. 对每条记录检查文件是否存在。
  5. 文件存在时提供“打开 / 打开目录 / 复制路径”。
  6. 文件不存在时提供“从列表移除”。
- 分支流程：
  - 空记录：显示空状态。
  - 文件存在：显示基础元数据和操作按钮。
  - 文件不存在：显示失效状态和移除按钮。
- 异常流程：
  - JSON 读取失败：显示加载失败，不崩溃。
  - 打开文件失败：显示失败反馈并写日志。
  - 打开目录失败：显示失败反馈并写日志。
  - 复制失败：显示失败反馈。
- 出口：
  - 点击关闭。
  - 操作完成后窗口保留。
- 状态变化：
  - 从列表移除会更新 JSON。
- 数据来源：
  - `QuickRecMetadata\recordings.json`。

### 10.4 托盘菜单入口

- 需求类型：前端 / UI。
- 触发条件：用户右键托盘图标。
- 入口：托盘空闲菜单。
- 主流程：
  1. 用户右键托盘。
  2. 菜单显示“最近录制”。
  3. 点击后打开最近录制窗口。
- 分支流程：
  - 录制中菜单是否显示最近录制由实现判断；建议至少空闲菜单显示。
- 异常流程：
  - 打开窗口失败：托盘通知提示失败，写日志。
- 实现约束：
  - 不影响现有录制、设置、诊断菜单。

### 10.5 结果条入口

- 需求类型：前端 / UI。
- 触发条件：录制成功保存。
- 入口：录制完成结果条。
- 主流程：
  1. 结果条显示“查看最近录制”。
  2. 用户点击后打开最近录制窗口。
  3. 新录制记录应出现在列表顶部。
- 异常流程：
  - 如果历史记录写入失败，仍可打开窗口，但可能显示旧记录或空状态。
- 实现约束：
  - 不移除现有打开文件 / 打开目录能力。

### 10.6 从列表移除

- 需求类型：前端 / 数据 / 文件。
- 触发条件：文件不存在或用户希望清理记录。
- 入口：最近录制窗口中的“从列表移除”按钮。
- 主流程：
  1. 用户点击“从列表移除”。
  2. 从 JSON 中删除该条记录。
  3. UI 从列表移除该条。
  4. 写入日志。
- 分支流程：
  - 文件存在时是否显示移除按钮可由实现决定；v1.5 至少在文件不存在时显示。
- 异常流程：
  - 写入失败：显示失败反馈，保留 UI 记录。

### 10.7 WGC / 新捕获后端 spike

- 需求类型：技术研究 / 文档 / 验证。
- 触发条件：进入 v1.5 技术研究任务。
- 入口：独立研究文档。
- 主流程：
  1. 评估 WGC 或其他系统级捕获后端是否能原生捕获光标。
  2. 评估普通窗口、最大化窗口、DPI 缩放场景。
  3. 评估 PyInstaller 打包影响。
  4. 输出研究记录。
- 分支流程：
  - 如无法实际验证，记录原因和待验证环境。
- 异常流程：
  - 不允许把未验证后端接入默认录制链路。
- 交付：
  - `doc/releases/v1.5/capture-backend-spike.md` 或等价研究文档。

### 10.8 Full 原型同步

- 需求类型：文档 / 原型 / UI。
- 触发条件：PRD 完成后。
- 入口：`doc/prototypes/product-prototype/full.html`。
- 主流程：
  1. 在 Full 原型中增加“最近录制”或突出素材入库入口。
  2. 原型展示可用记录和文件缺失记录。
  3. 原型提供打开、打开目录、复制路径、从列表移除等模拟操作。
- 异常流程：
  - 原型不调用真实文件系统。
- 验收：
  - 浏览器打开无空白。
  - 页面可切换到最近录制相关内容。

## 14. 风险与开放问题

### 14.1 风险

| 风险 | 影响 | 应对 |
| --- | --- | --- |
| 移除自绘光标后教程录制缺少光标 | 用户可能觉得操作不够清楚 | 明确这是质量取舍，后续通过 WGC 等原生捕获方案解决 |
| 历史记录写入失败影响录制保存 | 破坏核心路径 | 历史记录必须非阻塞，失败只写日志 |
| 最近录制入口膨胀成完整工作台 | v1.5 范围失控 | 只做最近 50 条，不做项目库和剪辑 |
| 保存路径变化导致历史分散 | 用户可能只看到当前保存目录历史 | v1.5 明确按当前保存目录读取历史，后续再评估跨目录索引 |
| JSON 文件损坏 | 最近录制无法读取 | 显示加载失败，不影响录制，日志可诊断 |
| WGC spike 变成主链路替换 | 回归风险高 | 明确不新增用户可见开关，不替换默认后端 |

### 14.2 当前开放问题

以下问题不阻塞 v1.5 PRD，但进入实现前建议在 dev plan 中明确：

- 最近录制窗口使用独立 `QDialog` 还是复用现有结果条组件。
- 历史记录写入由 `RecorderManager` 回调触发，还是由 `RecordingWorkflow` / `main.py` 统一协调。
- `duration_sec` 是否在 v1.5 必须准确获取；若成本较高，可允许为 null。
- 文件存在状态是打开窗口时即时检查，还是读取 JSON 时就更新 status。
