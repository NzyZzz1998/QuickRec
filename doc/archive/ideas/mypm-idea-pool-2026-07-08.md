# /idea 需求池 - 2026-07-08

## 总体判断

本需求池来源于 QuickRec 项目 review，当前证据主要来自本地代码、README、PRD、release notes、分支状态和历史验证记录。它们是内部项目证据，不等同于真实用户反馈，因此除 CI 覆盖、文档一致性这类工程治理项外，其他候选点都不能直接当成确定需求。

当前 QuickRec 处于两个并行阶段：

- **Full v1.4**：稳定性与工程化收口完成，后续更适合做兼容性、诊断、捕获后端研究。
- **Lite v0**：刚拆出轻量分支，当前最重要的是保持轻量边界、降低维护混淆、保证分支质量门槛。

结论上，最应该马上处理的是 **CI 覆盖 Lite 分支** 和 **README / 文档状态一致性治理**；最适合进入 `/prd` 的是 **诊断日志 / 错误导出能力**；需要继续验证的是 **Lite 底层代码隔离/删除**、**Lite 打包体积优化**、**Full 捕获后端升级**。

## 需求详情

### IDEA-001 Lite v0.1 是否需要隔离或删除区域/窗口录制底层代码

- 原始输入：Lite v0.1 是否需要隔离或删除区域/窗口录制底层代码。
- 输入类型：技术债 / 产品边界 / 方案。
- 产品问题：Lite 的用户可见入口已经裁剪为全屏录制，但底层仍保留区域/窗口录制能力。问题不是“代码多不好”，而是后续 Lite 维护时是否会因 Full 能力残留导致入口误回归、测试混淆、依赖难裁剪。
- 证据盘点：
  - 项目证据：`src/main.py` 仍有 `_on_start_region`、`_on_start_window`；`RecordingWorkflow` 和 `RecorderManager` 仍保留 `start_region` / `start_window`。
  - 文档证据：`doc/lite/release-notes-lite-v0.md` 明确写明 Lite v0 暂不删除底层代码，只移除用户可见入口。
  - 反证：Lite v0 刚完成发布收口，底层代码保留降低了大规模删除引入回归的风险。
- 证据等级：中。证据来自代码与文档，能证明“边界混杂存在”，但还没有证明它已经造成真实 bug 或维护成本。
- 价值判断：待验证偏中价值。它有助于 Lite 长期轻量，但现在直接删除可能过早。
- 当前阶段：Lite v0 发布后维护期，适合先评估，不适合马上大删。

总分：28/40，部分通过。证据强度为 3 分，允许进入小范围验证；但成本可控性只有 3 分，不建议直接删除。

- 低分项：用户痛点不是直接痛点；删除成本和回归风险仍在。
- 结论：继续验证，不直接删除。
- 最小下一步：做 Lite v0.1 三案对比：
  - A. 保留底层代码，只加强入口测试。
  - B. 将区域/窗口录制隔离为 Full-only 模块，Lite 不导入。
  - C. Lite 分支删除区域/窗口底层代码。
  对比体积、测试改动、回归风险、后续维护成本后再决定是否进入 `/prd`。

### IDEA-002 CI 是否需要覆盖 lite-master / lite-test 分支

- 原始输入：CI 是否需要覆盖 lite-master / lite-test 分支。
- 输入类型：工程治理 / 质量门槛。
- 产品问题：Lite 已经拆成独立分支并发布 `lite-v0`，但 CI 触发分支仍主要覆盖 Full 主线，会导致 Lite 后续提交缺少自动化质量门槛。
- 证据盘点：
  - 项目证据：`.github/workflows/ci.yml` 的 push 分支包含 `master`、`v1.4-engineering-baseline`，PR 目标包含 `master`。
  - 项目状态证据：本地和远端已有 `lite-master`、`lite-test`。
  - 反证：若 Lite 分支长期很少变更，可以降低触发频率，但不能完全没有质量门槛。
- 证据等级：强。分支策略和 CI 配置之间的不一致可以直接从仓库验证。
- 价值判断：高价值。它直接降低 Lite 分支回归风险，成本小、收益明确。
- 当前阶段：立即可做。
- 压力测试：跳过完整压力测试。理由：证据强、成本低、直接服务发布质量，且不新增用户功能复杂度。
- 结论：现在直接做。
- 最小下一步：更新 `.github/workflows/ci.yml`，让 push 覆盖 `master`、`test`、`lite-master`、`lite-test`，PR 至少覆盖 `master` 和 `lite-master`。如果担心 CI 成本，可以先覆盖 `lite-master` 和 `lite-test` 的默认测试，不跑硬件/打包 marker。

### IDEA-003 是否需要做诊断日志 / 错误导出能力

- 原始输入：是否需要做诊断日志 / 错误导出能力。
- 输入类型：稳定性 / 信任 / 支持能力。
- 产品问题：录屏工具失败高度依赖本机环境，包括 dxcam、FFmpeg、音频设备、权限、显示器、DPI、特殊窗口。当前日志分散在运行输出中，用户遇到无法录制或无法播放时，需要手动复制日志，不利于定位。
- 证据盘点：
  - 项目证据：`logging.basicConfig` 在入口初始化，录制、音频、FFmpeg、窗口诊断都有日志，但没有统一导出入口。
  - 历史证据：v1.4 过程中出现过 FFmpeg 启动失败、cv2 缺失导致 0 帧、音频混合失败、窗口录制光标/移动问题等。
  - 文档证据：v1.4 已把特殊窗口诊断、音频自检、本地硬件验收写入后续补充方向。
  - 反证：Lite 用户可能不希望看到复杂诊断 UI；做成诊断中心会偏重。
- 证据等级：中。内部测试和历史日志足以证明定位成本存在，但还没有真实用户反馈规模。
- 价值判断：中高价值。它不一定直接提升录制效果，但能显著提升出错后的信任感和定位效率。
- 当前阶段：适合进入 `/prd`，但必须做最小版本。

| 用户痛点强度 | 4 | 录制失败、无法播放、音频失败会直接损害信任。 |
| 目标用户/场景清晰度 | 4 | 用户在录制失败、保存失败、音频异常、窗口录制失败后需要导出诊断信息。 |
| 证据强度 | 3 | 有历史日志和测试问题，缺少外部用户规模数据。 |
| 频率/紧迫性 | 3 | 不是每天发生，但一旦发生影响严重。 |

| 成本可控性 | 4 | 最小版本可只做日志文件、错误摘要、复制/打开日志目录。 |

总分：31/40，部分通过，接近推进候选。证据强度 3 分，因此可以进入小范围 `/prd`，但不建议一开始做复杂诊断中心。

- 低分项：真实用户反馈规模不足；频率不确定。
- 结论：进入 `/prd`，限定为最小诊断导出能力。
- 最小下一步：写轻量 PRD，范围只包含：
  - 保存最近一次启动/录制关键日志。
  - 设置页或托盘增加“打开日志目录 / 复制诊断信息”。
  - 诊断内容包含版本、分支、系统、FFmpeg 路径、录制模式、音频模式、最后错误。
  - 不做完整诊断中心、不做云上传。

### IDEA-004 Lite 打包体积是否需要继续优化

- 原始输入：Lite 打包体积是否需要继续优化。
- 输入类型：工程优化 / 产品气质 / 技术债。
- 产品问题：Lite 定位是轻量录屏工具，但当前稳定包体积约 `257.89MB`，与“轻量”气质不完全一致。问题不是体积本身，而是体积是否会影响下载、分发、用户信任，以及是否会迫使 Lite 保留不必要依赖。
- 证据盘点：
  - 项目证据：README 和 release notes 均记录 Lite v0 体积 `257.89MB`。
  - 技术证据：主要来源是 FFmpeg、OpenCV/cv2、PyQt5、NumPy、Python runtime。
  - 历史反证：移除 cv2 曾导致 dxcam 捕获失败，说明不能以稳定性为代价硬砍依赖。
  - 用户证据：暂无用户因体积放弃下载或安装的反馈。
- 证据等级：中。体积事实明确，但用户影响未验证。
- 价值判断：中价值。值得持续优化，但不应阻断 Lite v0，也不应牺牲稳定性。
- 当前阶段：Lite v0.1+ 持续优化，不建议当前作为最高优先级。

#### 压力测试

| 维度 | 分数 | 证据/理由 |
| --- | ---: | --- |
| 用户痛点强度 | 3 | 体积偏大可能影响下载和轻量感，但暂无用户放弃证据。 |
| 目标用户/场景清晰度 | 3 | 目标是首次下载、分发、低配置用户；场景清楚但证据不足。 |
| 证据强度 | 3 | 有体积报告和依赖构成，缺用户影响证据。 |
| 频率/紧迫性 | 2 | 只在下载/安装时影响，录制核心路径不受影响。 |
| 核心目标贡献 | 4 | Lite 的产品气质要求轻量，长期相关。 |
| 差异化/替代方案 | 3 | 可用压缩、依赖替换、后端替换，但都有风险。 |
| 成本可控性 | 4 | 可按实验逐项推进，失败可回滚。 |
| 验证速度 | 5 | 包体积和 smoke 测试很快能验证。 |

总分：27/40，部分通过。适合继续做实验，不适合直接承诺体积目标或强行砍依赖。

- 低分项：用户影响证据不足；紧迫性不高。
- 结论：继续验证。
- 最小下一步：
  - 延续 package size report。
  - 每次只做一个变量实验：FFmpeg 精简包、OpenCV headless、Qt 插件排除、Full-only 模块隔离。
  - 每个实验必须跑硬件 smoke，不能只看体积。
  - 继续维持“低于 200MB 是目标，不是阻断项”。

  - 检查 README、release notes、progress、test cases、PRD 中的版本、分支、tag、打包路径、体积、验证状态。

1. **诊断日志 / 错误导出能力**
   - 理由：稳定性价值明确，历史问题足够支撑最小版本。
   - PRD 范围建议：只做“打开日志目录 / 复制诊断信息 / 最近错误摘要”，不要做完整诊断中心。

### 继续验证

1. **Lite v0.1 隔离或删除区域/窗口录制底层代码**
   - 验证方式：三案对比 + 入口测试 + 打包体积对比。

2. **Lite 打包体积继续优化**
   - 验证方式：逐项实验，不牺牲 dxcam/cv2 稳定性。

3. **Full 捕获后端升级 / WGC 研究**
   - 验证方式：research spike，不改主线。

### 暂不做

当前没有候选点被判定为永久不做，但以下做法现在不该做：

- 不要在 Lite v0.1 直接大规模删除录制底层代码。
- 不要为了体积移除 cv2 或外置 FFmpeg，除非硬件 smoke 证明替代方案稳定。
- 不要把 WGC 捕获后端升级直接写成 Full 必做需求。
- 不要把诊断能力做成复杂诊断中心、云上传或后台系统。

## 后续承接建议

如果下一步要进入开发，建议顺序是：

1. 先做 CI 分支覆盖和文档一致性治理。
2. 再对诊断日志 / 错误导出写 `/prd`。
3. 并行做 Lite 底层隔离、体积优化、WGC 的 research brief，不进入正式开发。
