# QuickRec v1.4-M5 替代捕获后端研究记录

> 状态：研究记录  
> 适用范围：v1.4-M5 特殊窗口兼容性诊断  
> 结论：v1.4-M5 不强制替换 dxcam 捕获链路；先补齐失败 reason、日志诊断和回归保护，再评估替代方案。

## 1. 背景

v1.4 已完成全屏、区域、常规窗口录制的稳定性收口，并验证 dxcam 默认捕获链路依赖 `cv2`。在特殊窗口场景中，游戏全屏、UWP、DWM 自定义渲染窗口仍可能出现无法获取窗口区域、捕获内容异常或捕获后端启动失败。

M5.1 的目标不是立即新增特殊窗口兼容能力，而是先做到：

- 失败不阻塞主线程。
- 失败不导致软件崩溃。
- 日志包含 hwnd、title、mode、rect、foreground_result、failure_reason。
- 普通全屏、区域、常规窗口录制不回退。

## 2. 当前方案

当前实现继续使用 dxcam 作为主捕获链路，并新增结构化窗口诊断：

| 字段 | 说明 |
| --- | --- |
| `reason` | `unsupported_window`、`rect_unavailable`、`foreground_denied`、`capture_backend_failed` 等失败原因 |
| `hwnd` | 窗口句柄 |
| `title` | 窗口标题 |
| `mode` | 当前录制模式 |
| `stage` | 失败阶段 |
| `rect` | 捕获区域，无法获取时为 `None` |
| `foreground_result` | 置前台结果或 `not_attempted` |

## 3. 替代方案评估方向

| 方向 | 适用场景 | 依赖变化 | 体积影响 | 迁移风险 |
| --- | --- | --- | --- | --- |
| 系统级窗口捕获后端 | 特殊窗口、系统合成窗口、部分自定义渲染窗口 | 可能新增 Windows API 封装或桥接依赖 | 中，需要验证打包 DLL 和运行时组件 | 高，窗口选择、权限、DPI、帧格式都需要重建 |
| 应用帧源捕获策略 | 对特定应用或特定渲染路径做兼容 | 可能需要更强的窗口元数据识别 | 中到高，依赖不确定 | 高，兼容性收益可能只覆盖少数应用 |
| GPU 复制路径替代 | 解决 dxcam 某些帧格式或移动窗口场景 | 可能替换部分帧处理链路 | 中，可能影响 cv2 / numpy 依赖边界 | 中到高，性能和颜色格式需要大量验证 |
| 失败诊断增强 | 不提升捕获成功率，但提升可定位性 | 无新增运行时依赖 | 低 | 低，是当前 M5.1 首选路径 |

## 4. 推荐结论

v1.4-M5 阶段优先保留 dxcam 主链路，不引入替代捕获后端。原因：

1. 当前全屏、区域、常规窗口录制已经通过 v1.4 验证，替换捕获后端会放大回归风险。
2. 特殊窗口兼容性收益不确定，不能以破坏普通录制稳定性为代价。
3. 打包体积仍处于稳定性优先阶段，新增捕获依赖会影响体积优化判断。
4. 结构化诊断可以先解决“失败不可定位”的问题，为后续替代后端评估提供样本。

后续若继续推进，应先建立专门的硬件测试样本集，再在独立分支验证替代捕获后端，不建议直接在 v1.4 主线替换。

## 5. 验收要求

- 替代方案研究不得影响现有 dxcam 主链路。
- 任一候选方案进入实现前，必须先通过全屏、区域、窗口录制回归。
- 任一候选方案进入实现前，必须记录打包体积变化。
- 任一候选方案进入实现前，必须验证音频混流不受影响。
