# QuickRec - 轻量级录屏工具 PRD

## 1. 产品概述和目标用户

### 1.1 产品概述

QuickRec 是一款**轻量级、零配置**的 Windows 桌面录屏工具。用户打开即用，一键开始录制，自动保存到指定文件夹，无需登录、无需云同步、无多余功能。核心定位：比系统自带录屏更灵活，比专业录屏软件更简单。

### 1.2 目标用户

| 用户类型 | 使用场景 | 频率 |
|---------|---------|------|
| 办公人群 | 录制操作教程、bug 复现步骤、会议要点 | 每周 2-5 次 |
| 教师/培训师 | 录制课件讲解、操作演示 | 每周 1-3 次 |
| 开发者 | 录制 bug 复现、功能演示 | 偶尔 |
| 普通用户 | 录制游戏片段、生活记录 | 偶尔 |

### 1.3 产品目标

- **3 秒内开始录制**：从点击录制到实际录制不超过 3 秒
- **零学习成本**：首次使用无需教程即可完成录制
- **安装包 < 50MB**：轻量不占资源

##### 3.5.5.3 架构继续拆分

v1.4 已完成录制工作流、状态机、事件模型等初步解耦。后续补充需求不限定具体模块命名，但要求继续按职责边界收敛复杂度：

| 方向 | 目标 |
|------|------|
| 应用装配边界 | 降低入口文件对托盘、热键、录制、通知、配置的直接编排 |
| 录制生命周期边界 | 将开始、暂停、恢复、停止、取消、保存完成等状态转换保持在统一工作流内 |
| 通知与提示边界 | 将系统通知、错误提示、托盘反馈与录制核心逻辑隔离 |
| 窗口录制边界 | 将窗口定位、窗口移动检测、区域换算、异常窗口判断与通用录制流程隔离 |
| 资源释放边界 | 明确 FFmpeg、dxcam、音频线程、临时目录、系统计时器等资源的创建与释放责任 |

验收标准：
- 新增或修改逻辑时不得重新扩大 `main.py` 和录制管理类的职责。
- 关键路径必须有单元测试或可替代的集成测试覆盖。
- UI 层不应直接读取录制核心的私有字段。

##### 3.5.5.4 本地硬件验收补充

GitHub CI 无法覆盖真实桌面、真实音频设备、dxcam 捕获链路和打包产物运行环境，因此 v1.4 后续补充必须提供本地硬件验收能力。

- 硬件脚本作为发布前必跑项，而不是普通开发的默认测试项。

##### 3.5.5.5 打包体积优化补充

v1.4 已验证移除 cv2 会破坏 dxcam 默认捕获链路，因此后续体积优化不得以破坏录制稳定性为代价。该项不设置固定体积上限，要求先建立体积构成分析，再逐项评估可行优化。

| 优化方向 | 要求 |
|---------|------|
| 体积构成分析 | 输出 PyInstaller 产物中 FFmpeg、OpenCV、Qt、Python 运行时、依赖库的体积占比 |
| FFmpeg 优化 | 评估更小 FFmpeg 构建或裁剪方案，但不得外置 FFmpeg |
| OpenCV 优化 | 评估 `opencv-python-headless`、dxcam 处理器替代、按需依赖等方案 |
| Qt/PyInstaller 优化 | 继续排除未使用 Qt 模块、插件、DLL、PIL 编解码器和其他无关资源 |
| lite 分支依赖优化 | 将体积优化与未来 lite/full 分支依赖边界结合，不在 v1.4 强行拆分 |

验收标准：
- 每次打包必须记录产物体积和主要依赖构成。
- 任何体积优化都必须通过全屏、区域、窗口录制和音频混流回归。
- 不以压缩包外置依赖替代单目录可运行产物。

##### 3.5.5.6 lite/full 未来规划

lite/full 在 v1.4 只做规划，不做实际分支拆分，不改变当前主线功能边界。

| 分支方向 | 产品定位 | 功能边界 |
|---------|---------|---------|
| QuickRec Lite | 轻量级录屏工具，长期保持简单、低心智负担和较小依赖 | 默认仅保留全屏录制；区域录制作为未来可选能力；不规划窗口录制；保留托盘 UI；保留当前音频能力但隐藏高级配置 |
| QuickRec Full | 创作者工作台，面向录制、管理、轻编辑和导出的完整工作流 | 录制历史、项目素材管理、轻量剪辑、导出队列、质量诊断中心、模板与预设、更多兼容性策略 |

规划原则：
- lite 和 full 后续应作为不同演进方向，不要求在完全一致的项目内长期共存。
- lite 优先减少依赖、降低入口复杂度、保证快速录制。
- full 优先围绕创作者工作台扩展能力，不把复杂功能回灌到 lite。
- v1.4 阶段只沉淀边界和技术准备，不做用户可见的 lite/full 切换。

### 5.3 核心设计决策

| 决策 | 原因 |
|-----|------|
| FFmpeg pipe 实时编码（v1.3） | dxcam 截图通过 subprocess pipe 直接送 FFmpeg 编码为 H.264，无需 JPEG 临时文件后编码；停止录制后文件即时可用，无等待时间 |
| H.264 固定参数 | 内部固定 CRF 23 + preset superfast + tune zerolatency，不暴露给用户；兼顾画质、实时写入和停止时 flush 延迟，1080p/30fps 约 8MB/分钟 |
| 窗口丢失简化处理（v1.3） | 窗口关闭→自动停止并保存（无弹窗）；窗口最小化→自动暂停+托盘通知（用户恢复窗口后点托盘继续）；避免 QMessageBox 在托盘应用中不显示的问题 |
| dxcam 区域缓存 | update_region() 对比新 region 与缓存值，相同则跳过重建；200ms 间隔更新窗口位置 |
| 临时文件缓存方案（v1.2 遗留，v1.3 废弃） | v1.2 使用 JPEG 写临时文件+后编码方案；v1.3 改为 FFmpeg pipe 实时编码，去除临时文件环节 |
| 非阻塞 stop() | stop() 立即返回，编码完成在后台线程处理 |
| 三处信号桥 | pyqtSignal 机制将 pystray/pynput/编码线程的回调安全转发到 Qt 主线程 |

### 5.5 v1.4 架构优化方向

v1.4 的架构优化不改变用户操作路径，只降低模块耦合和维护成本。具体模块命名可在技术设计阶段细化，但 PRD 要求达到以下边界目标：

| 模块方向 | 目标职责 | 验收要求 |
|---------|---------|---------|
| 应用控制器 | 负责 Qt 应用生命周期、托盘启动、退出收尾 | `main.py` 不再承载完整业务流程，入口职责清晰 |
| 录制工作流 | 串联全屏/区域/窗口录制的开始、倒计时、暂停、停止、取消、保存完成 | 三种录制模式复用同一套状态流，减少重复连接和回调 |
| 录制状态机 | 管理 IDLE/RECORDING/PAUSED/STOPPING/SAVING 等状态转移 | 非法状态转移有明确返回或错误事件，可单元测试 |
| 通知服务 | 统一托盘通知、保存结果、失败提示 | FFmpeg、磁盘、窗口失败等提示文案集中管理 |
| 窗口录制服务 | 处理窗口选择、置前台、窗口丢失、边框高亮协作 | 特殊窗口失败不阻塞主线程、不崩溃 |
| 资源生命周期管理 | 管理 dxcam、FFmpeg、音频、临时目录、系统计时器 | 连续录制和退出流程可测试、可诊断 |

必须移除 UI 层对 recorder 私有字段的直接访问。保存完成、录制失败、窗口丢失、状态变化等信息必须通过公开事件、回调或信号接口传递。

### 6.5 磁盘空间预警规格（v1.3 P9）

| 规格项 | 值 | 说明 |
|-------|-----|------|

## 8. 风险与应对

| 风险 | 影响 | 应对 |
|-----|------|------|
| FFmpeg 子进程崩溃 | 正在录制的视频可能损坏或丢失 | v1.3：FFmpeg pipe 写临时文件（session_dir/video.mp4），停止后 shutil.move 到最终路径；atexit 与 cleanup_stale 兜底清理崩溃残留 |
| FFmpeg pipe stderr 缓冲满死锁 | 进程卡死、停止时 wait 超时被 kill | ✅ 已解决（Bug #58）：`stderr=subprocess.DEVNULL` 丢弃进度日志，避免 64KB 缓冲满阻塞 FFmpeg |
| H.264 编码 CPU 占用 | 实时编码比 JPEG 缓存方案 CPU 占用更高 | CRF 23 + preset superfast + tune zerolatency 平衡画质和性能；lookahead=0 降低停止 flush 延迟 |
| 窗口丢失处理 | 窗口关闭/最小化后录制行为 | ✅ 已解决：关闭→自停保存+托盘通知；最小化→录制线程同步暂停（clear _resume_event），用户恢复后点继续 resume |
| 特殊窗口（游戏/UWP）置前台失败 | SetForegroundWindow 被 Windows 前台锁拒绝，录制背景 | ✅ 已解决（Bug #51/#57）：Alt 键模拟释放前台锁 + BringWindowToTop；主线程异步化避免长阻塞闪退 |
| Windows 窗口枚举兼容性 | 不同 Windows 版本窗口列表获取方式不同 | 使用 Win32 API EnumWindows，过滤不可见/系统窗口 |
| 打包体积过大 | 安装包因 OpenCV 和 Qt 插件膨胀 | ✅ 已解决：excludes + 二进制过滤排除 Qt DLL/ANGLE/OpenCV videoio/PIL 编解码器；最终 259MB（核心依赖不可压缩） |
| DPI 缩放 | 4K/150% 缩放下 UI 元素错位 | Qt Fusion 样式 + 手动 DPI 适配 |
| 倒计时与录制线程时序 | 倒计时结束与录制开始的原子性 | ✅ 已解决：倒计时在 Qt 主线程展示，结束后通过信号触发录制线程 |
| 鼠标点击高亮性能 | 每帧绘制高亮动画增加编码开销 | ✅ 已解决：高亮仅屏幕叠加层，不渲染到视频帧 |
| 开机自启注册表操作 | 写入注册表可能触发安全软件警告 | ✅ 已解决：仅写 HKCU\Run，无需管理员权限 |
| v1.4 测试先行成本 | 先修测试会推迟架构和运行时问题的实际修复 | 采用 M1→M2→M3→M4 顺序，先建立可信回归网，再进行高风险重构 |
| v1.4 架构解耦回归 | 拆分 `main.py` / `RecorderManager` 可能影响录制流程 | M1 必须先完成；状态机、公共事件接口、三种录制模式都要有回归测试 |
| v1.4 打包体积未低于 `< 200MB` | 不允许外置 FFmpeg；移除 cv2 会破坏 dxcam 默认捕获链路 | 稳定性优先恢复 cv2，记录体积构成；lite 分支再评估更小 FFmpeg / headless OpenCV / 捕获后端替换 |
| CI 无法覆盖真实桌面录制 | GitHub Actions 环境缺少 Windows 桌面、音频、dxcam 真实硬件上下文 | CI 覆盖纯逻辑和可模拟路径；硬件/系统集成测试用 marker 单独执行，并纳入发布前手工 checklist |
| 特殊窗口兼容性边界 | 游戏/UWP/DWM 自定义渲染窗口仍可能不可录制 | v1.4 不承诺新增兼容能力，只要求失败可诊断、提示稳定、不崩溃 |
